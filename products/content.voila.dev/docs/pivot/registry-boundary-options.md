# Pivot Рђћ The Package / Registry Boundary

> Status: **reflection / pre-decision**. Engine depth is already settled:
> the core is **Full Effect platform** (`effect/Schema`, `@effect/sql`,
> `@effect/platform` `HttpApi`, `@effect/rpc`, `@effect/cli`, `Layer`-based
> adapters). The frontend stays **TanStack + React**.
>
> The open question this doc resolves: **what ships as a versioned npm
> dependency vs. what is vended, shadcn-style, into the consumer's repo as
> code they own.**

---

## Why this is the decision that everything else hangs on

Two pivots are in flight at once, and they pull in opposite directions:

- **Effect-only engine** wants a *deep, opaque, versioned dependency*. You do
  not want users editing `Effect<A, E, R>` resolver internals, `HttpApi`
  derivation, or `@effect/sql` query compilation. Effect is powerful precisely
  because it is encapsulated.
- **shadcn "you own your code"** wants the *opposite*: real source files
  copied into the user's repo, no black box, fully editable.

The art is drawing the line so that **the Effect complexity stays behind a
dependency the user never has to understand, while everything the user
legitimately wants to customize is code they own.** Draw it in the wrong place
and you either (a) recreate today's black box, or (b) force every consumer to
learn Effect to run a CMS Рђћ which would violate the DX north star
("no new concepts beyond TanStack").

## The layer cake

Every option below is just a different horizontal cut through these layers.
Layers are ordered from "deep engine" (top) to "surface you touch daily"
(bottom).

| #   | Layer                                    | Tech                                  | Today |
| --- | ---------------------------------------- | ------------------------------------- | ----- |
| L1  | Schema runtime (fields, validate, infer) | `effect/Schema` + annotations         | `@voila/content-schema` |
| L2  | Resolvers (query/mutation/RBAC/hooks)    | `Effect`, `Layer`                     | `@voila/content/server` |
| L3  | Adapters (db/storage/queue/cron/auth)    | `@effect/sql-*`, `Layer`              | `database`, `auth` |
| L4  | HTTP API definition + derived handlers   | `@effect/platform` `HttpApi`          | hand-rolled handlers |
| L5  | Typed client                             | derived from `HttpApi`                | `@voila/content/client` |
| L6  | Server route files (mount points)        | TanStack Start server routes          | **virtual** (vite plugin) |
| L7  | Admin route files (`/admin/*`)           | TanStack Router                       | **virtual** (vite plugin) |
| L8  | Admin shell (layout, sidebar, branding)  | React                                 | `content/admin` |
| L9  | Data views (list/detail)                 | TanStack Table + Query                | `content/admin` |
| L10 | Forms                                    | TanStack Form                         | `content/admin` |
| L11 | Field widgets + theme/tokens             | React + shadcn + Tailwind v4          | `content/admin/widgets`, `@voila/ui` |

> Today **L6 and L7 are *virtual*** Рђћ generated by the `voila()` vite plugin
> and invisible to the consumer. That is the specific "magic" the pivot is
> reacting against.

## Option A Рђћ Engine as dependency, the whole head is vended

**Dependency (npm):** L1РђЊL5 Рђћ the headless Effect engine, adapters, the
`HttpApi`, and the derived typed client. No React anywhere.

**Vended by the registry (you own):** L6РђЊL11 Рђћ a *thin* server route file that
mounts the engine handler, the admin route tree, the shell, tables, forms,
widgets, theme.

```
npm i:        @voila/content        (Effect engine, headless, semver'd)
voila add:    app/routes/admin/*     (TanStack route files Рђћ yours)
              components/admin/*     (shell, tables, forms Рђћ shadcn-style)
              widgets/*              (field widgets Рђћ yours)
              server/voila.ts        (3-line mount of the engine handler)
```

**Pros**
- The cut lands exactly on the "framework I already know" seam: everything you
  own is **TanStack + React**; everything you don't is **Effect you never see**.
- Kills the virtual-route magic. L6/L7 become real files you can read, add
  middleware to, re-path, or extend with your own endpoints.
- Engine bugs and Effect upgrades ship centrally via npm semver; the head
  evolves at the user's pace (they copied it, they own the churn).
- Matches the proven shadcn shape almost exactly Рђћ the registry vends
  presentational + wiring code; the heavy logic stays a dep.
- `HttpApi` stays a dependency, so **OpenAPI, the typed client, and server
  validation regenerate for free** on upgrade.

**Cons**
- The vended surface is large (L6РђЊL11). The registry must model
  inter-component dependencies (a `posts-table` needs a `data-table`
  primitive) Рђћ more registry engineering than shadcn's mostly-flat catalog.
- "Update my CMS admin" is now a user-driven re-pull/merge, not an `npm
  update`. Needs a good `voila diff` / re-vend story.
- The engine Рєћ vended-code contract (typed client shape, widget props) becomes
  a public API you must keep stable across versions.

**Who it's for:** the default target user Рђћ a TanStack dev who wants a great
admin fast and the freedom to restyle/extend it without forking, and who
should never have to open an Effect file.

## Option AРђ▓ Рђћ Granular engine packages, extend via `Layer` (the AРєћB midpoint)

Same **boundary** as A Рђћ the head (L6РђЊL11) is vended, the engine is a
dependency, you never *have* to open an Effect file. The one change: **the
engine is not one opaque package but a set of small packages whose Effect
`Service`/`Layer` contracts are public API.** You extend by *providing a
different `Layer`*, not by owning source.

This is the natural Effect grain. In Effect you don't customize a library by
editing it Рђћ you customize it by supplying a different implementation of a
`Service` interface. A monolithic engine (plain A) hides those seams; granular
packages expose them. This is the "all the benefits of A, but I can still play
with Effect when I want to" path.

**Dependency (npm), decomposed:**
`@voila/content-schema` (L1) ┬и `@voila/content` (L2, resolver `Service`s) ┬и
`@voila/content-sql` + dialect layers `@voila/content-sql/{d1,pg,sqlite}` (L3) ┬и
`@voila/content-storage` (+ adapter layers) ┬и `@voila/content/server` (L4, the `HttpApi`) ┬и
`@voila/content/client` (L5).

**Vended (you own):** L6РђЊL11 Рђћ identical to A.

**Extension surface (opt-in, additive Рђћ nothing ejected):**
```ts
// Swap the database Рђћ provide a different Layer; the engine is untouched
ContentRuntime.pipe(Layer.provide(TursoLayer))        // instead of D1Layer

// Add a cross-cutting rule Рђћ wrap a Service, don't fork it
const AuditedMutations = Layer.effect(MutationService, /* tap the default */)

// Add your own endpoints, reusing the engine's Services
const MyApi = HttpApi.add(voilaApi, myGroup)           // composes, not replaces

// A bespoke field type = a Schema + a Layer, registered Рђћ no source owned
```

**Pros Рђћ everything A has, plus:**
- **Extensibility without ownership.** You reach B-grade customization (swap DB,
  intercept mutations, add endpoints, custom fields) by *composing `Layer`s*,
  while the engine source stays a versioned dependency that still ships you
  upgrades. No forking, no derived-code rot.
- **Effect stays opt-in.** Ignore every seam and AРђ▓ behaves exactly like A Рђћ the
  granularity is invisible until you reach for it.
- **Smaller footprint.** A consumer who never touches storage never pulls
  `@voila/content-storage`; tree-shaking is per-package.
- **Cleaner seams for the maintainer too** Рђћ explicit `Service` boundaries are
  easier to test and reason about than one mega-package.

**Cons:**
- More packages to version and document; each `Service` interface becomes public
  API you must keep stable. *Mitigated:* lock-step releases are already wired
  (`fixed: [["@voila/*"]]` in changesets), so they all move as one version.
- A consumer who *does* extend must learn Effect's `Service`/`Layer` model Рђћ but
  **only if they opt in**, unlike B which forces Effect on everyone.
- More dependency-graph discipline required (avoid cycles between
  core Рєћ http Рєћ client).

**Who it's for:** the default user *and* the power user Рђћ without making the
second group fork or the first group learn Effect.

## Option B Рђћ Thin core, fat registry (maximal ownership)

**Dependency (npm):** L1РђЊL3 only Рђћ schema runtime, resolver primitives, adapter
contracts. The irreducible Effect core.

**Vended (you own):** L4РђЊL11 Рђћ including the `HttpApi` definition **and the
derived handler wiring**, plus everything visual.

**Pros**
- Maximum ownership. Nothing about the API surface or server behavior is
  hidden; ideal for teams with compliance/audit needs or heavy server-side
  customization.
- The smallest possible dependency footprint to keep current.

**Cons**
- Forces consumers to **own `@effect/platform` and `Effect` code** to maintain
  their handlers. This directly violates "no concepts beyond TanStack" and
  imports the full Effect learning curve into every consuming team.
- You forfeit the biggest win of `HttpApi`-as-dependency: free OpenAPI/client
  regeneration and centrally-fixed handler bugs. Vended derived code rots.
- Largest maintenance + support surface for the maintainer: every consumer's
  handlers can diverge, so "works on my repo" support explodes.

**Who it's for:** power users / platform teams who *want* to live in Effect.
A minority of the addressable market.

---

## Option C Рђћ Engine + API + server are the dependency; only the visual head is vended

**Dependency (npm):** L1РђЊL7 Рђћ everything including server and admin route
registration (kept behind a lighter plugin, or thin generated files).

**Vended (you own):** L8РђЊL11 Рђћ shell, data views, forms, widgets, theme.

**Pros**
- Smallest jump from today's architecture; L1РђЊL7 stay roughly as built.
- Users get to restyle/replace the *visual* layer, which is 80% of real
  customization requests.

**Cons**
- The server/route layer (L6/L7) **stays magic** Рђћ which is the exact thing the
  pivot set out to remove. "You own your code" is only half-true.
- You keep shipping a runtime admin package (the black box you're trying to
  retire) *and* a registry Рђћ two UI delivery systems to maintain.
- Escape hatch for server behavior is "fork the dep," the worst-case DX.

**Who it's for:** a conservative half-step. Hard to justify given the stated
motivation for the pivot.

---

## Option D Рђћ Dependency by default, eject on demand (two-tier)

**Dependency (npm):** *everything*, including a batteries-included React admin
package Рђћ exactly like today's one-line install.

**Registry:** a **mirror of every component**, vended on demand, so any single
piece can be "ejected" into the user's repo to override the dependency's
version (via import aliasing / route shadowing).

**Pros**
- Best initial DX: one-line install, zero files to own until you want to.
- Ownership becomes *opt-in per component* Рђћ you only take on maintenance for
  the pieces you actually customize.

**Cons**
- You maintain **both** a full runtime library **and** a full registry mirror,
  kept in lockstep forever. Heaviest maintenance load of all four Рђћ untenable
  at 1 FTE.
- The override/resolution mechanics are genuinely hard with a TanStack
  file-based route tree (which file wins, how aliasing composes with SSR).
- Philosophically muddy: the default is still a black box, so the pivot's
  "you own your code" identity is diluted into an advanced feature.

**Who it's for:** large libraries with a team to maintain the dual system
(closer to a Next.js-scale operation). Not a 1-FTE project.

## Recommendation Рђћ **Option AРђ▓**

I'd commit to **AРђ▓: the same boundary as A Рђћ headless engine as the
dependency, the whole head vended Рђћ but with the engine decomposed into
granular packages whose `Service`/`Layer` seams are public API.** AРђ▓ keeps
every argument for A below intact (they share the boundary) and adds the one
thing monolithic A lacks: **B-grade extensibility without B's forced
ownership.** Given the engine is now Full Effect platform Рђћ a stack *built*
around `Layer` composition Рђћ shipping it as one opaque box would waste its
defining strength. AРђ▓ is A done the Effect way.

Argued through the three lenses you asked for (all hold for both A and AРђ▓):

### 1. Current TS ecosystem fit

- The shadcn registry pattern (a `registry.json` + source files + a CLI that
  resolves deps and rewrites imports) is now a **documented, stable, widely-copied
  convention** Рђћ not a bet. TanStack's own sensibility is drifting toward
  "you own the code." A puts voila squarely on that wave.
- `@effect/platform`'s `HttpApi` is *most* valuable as a dependency: it derives
  the server, the typed client (L5), and OpenAPI from one definition. That
  compounding leverage only works if the consumer keeps receiving it via semver
  Рђћ which is exactly A's boundary, and exactly what B throws away.
- Effect is a deliberately encapsulated runtime. The ecosystem treats it as an
  *engine*, not as code you scatter across app repos. A respects that grain; B
  fights it.

### 2. DX

- A is the only option that **preserves the DX north star** Рђћ "a senior dev
  scaffolds a CMS in an afternoon with no concepts beyond TanStack." Everything
  the user touches (L6РђЊL11) is TanStack + React + Tailwind. Effect lives behind
  the import and never surfaces in their editor.
- The escape hatch becomes *real*, not aspirational: today's table of
  overrides ("slug derived from title Рєњ `fields.slug({...})`") is replaced by
  "open the file the registry gave you and edit it." That's the strongest
  possible answer to "the lib abstracts too much."
- Replacing virtual L6/L7 with real TanStack Start route files is a **net DX
  upgrade even before customization**: the route tree is finally greppable,
  debuggable, and extensible with the user's own endpoints/middleware.

### 3. Ownership / global maintenance

- A draws the cleanest maintenance seam: **the engine is fixed centrally and
  shipped by semver; the head is owned by the user once vended.** Engine
  security/perf fixes reach everyone via `npm update`; the user absorbs only
  the churn of UI they chose to own.
- It is the **realistic load for 1 FTE**. B and D both make you support code
  that has diverged in every consumer repo (B: handlers; D: a full mirror kept
  in lockstep). C makes you maintain two UI systems. A makes you maintain *one*
  engine + *one* registry, with a clear contract between them.
- The one real cost Рђћ the engineРєћvended contract (typed-client shape, widget
  props) becoming a stable public API Рђћ is a cost you *should* pay anyway; it's
  what makes the thing a product rather than a pile of internals.

### 4. Effect grain (why AРђ▓ over plain A)

- Effect's whole extensibility model is **provide-a-`Layer`, not edit-the-code.**
  Plain A buries the `Service` interfaces inside one package, so the only
  escape hatch left for engine behavior is "eject the source" (B). AРђ▓ surfaces
  those interfaces as public API, so customization is *additive composition* Рђћ
  the idiomatic move Рђћ while the source stays a dependency.
- It turns the AРєћB tension into a **dial, not a fork**: 90% of users provide
  the default `Layer`s and never see Effect; the 10% who want to swap a DB,
  intercept a mutation, or add endpoints do so without owning a single engine
  file. Both groups upgrade via `npm update`.
- Granular packages are also the ecosystem norm for an Effect-based stack
  (Effect and TanStack both ship this way), so it reads as conventional, not
  exotic.

### The precise cut I'd draw inside AРђ▓

The elegant detail is **L6**: vend a *thin* server route file, not the handler
logic.

```ts
// app/server/voila.ts  Рђћ VENDED, you own this 3-line mount
import { handler } from "@voila/content/server"   // <- Effect engine (dep)
import config from "~/content.config"

export const voilaHandler = handler(config)        // add your middleware here
```

The user owns the **mount point** (path, auth middleware, rate limiting, extra
routes) while the **handler logic stays in the Effect dependency**. That single
seam is what makes A "you own your code" *and* "you never touch Effect" at the
same time Рђћ the two goals that started as a contradiction.

### What's left for B

AРђ▓ absorbs most of B's reason to exist: swapping a DB, intercepting mutations,
or adding endpoints no longer requires owning the engine Рђћ a `Layer` does it.
B is reduced to the genuine edge case of a team that wants the **handler/`HttpApi`
source itself** in their repo (audit, compliance, deep rewrites). Serve that as
an opt-in **`--eject-server` flag** on the registry CLI, never the default Рђћ
the minority gets it without taxing everyone else's maintenance.

## Implications for the revised roadmap (if AРђ▓ is chosen)

- The engine is **decomposed into granular Effect packages** with public
  `Service`/`Layer` seams Рђћ but consolidated to **8 product packages** (not one
  per layer): `@voila/content-schema` (L1 leaf), `@voila/content` (L2 resolvers +
  runtime + `/server` L4 + `/client` L5 + tasks + i18n fields), `@voila/content-sql`
  (L3, + `/d1` `/pg` `/sqlite` subpaths), `@voila/content-storage` (L3),
  `@voila/content-auth`, `@voila/content-mcp`, plus tooling `@voila/content-cli` +
  `@voila/content-registry`. Schema and auth stay separate (shared leaf / swappable
  heavy dep); tasks and i18n fold into `@voila/content`. Every Engine package is
  headless Рђћ React leaves all of them. See [Canon ┬Д3](./effect-architecture-canon.md).
- Each package's `Service` interface is **public, documented, and semver-stable**
  Рђћ the contract that makes opt-in `Layer` extension safe. (This is the new
  cost AРђ▓ takes on over A; lock-step changesets keep versions aligned.)
- `@voila/content-cli` is rebuilt on **`@effect/cli`** and grows the **registry
  commands** (`voila add <item>`, `voila diff`, `voila list`) Рђћ absorbing the
  old vite-plugin's job of materializing routes, but as real files. An opt-in
  `--eject-server` flag covers the B edge case.
- A new **registry package** (the `registry.json` + the vended source for
  L6РђЊL11) becomes a first-class deliverable Рђћ likely `@voila/content-registry` plus a
  hosted manifest the CLI reads.
- `@voila/content/client` (L5) is **derived from `HttpApi`** rather than
  hand-written; it may collapse into `@voila/content/server`'s exports.
- `@voila/ui` + `content/admin/*` components become **registry items**, not a
  shipped runtime Рђћ the "salvage the UI" path for the existing M0РђЊM2 React work.
- The `voila()` vite plugin's virtual-route responsibility is **retired**;
  routes are vended files. The plugin may survive only for HMR/config glue.
