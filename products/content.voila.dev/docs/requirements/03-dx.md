# 03 — Developer Experience

> The most important doc in this repo.

Every other decision in this project is downstream of the answer to: *what does it feel like to use?*

## The DX north star

> A senior dev should be able to scaffold a production CMS for a real product in **one afternoon**, without learning any new concepts beyond their existing TanStack knowledge, and without ever leaving their editor.

If that's not true on any release, we cut features.

## Five DX commitments

### 1. One file, one source of truth

`content.config.ts` is the only thing you maintain. From it we derive:

| Artifact            | Where it lives                      |
| ------------------- | ----------------------------------- |
| Admin UI            | `@voila/content` runtime            |
| REST/RPC API        | `@voila/content` runtime            |
| Typed client        | inferred from `typeof contentConfig`|
| DB schema           | `bunx voila migrate`                |
| Validators          | inferred from field constructors; any [Standard Schema](https://standardschema.dev/) library — Zod by default |
| MCP tools/resources | `@voila/mcp` reads the config       |
| Search index        | derived from `field.searchable`     |
| Webhook payloads    | inferred doc type                   |

You never edit two places to add a field. Adding `tags: fields.array(fields.string())` to the config makes the column appear in the admin, in the API, in the typed client, in the MCP server, and in the search index. Period.

### 2. Type inference, never codegen

We don't ship a `voila generate` step. Types come from `as const` + conditional types:

```ts
const config = defineContent({ collections: [posts, authors] } as const)

type Post = InferDoc<typeof config, 'posts'>
//   ^ { id: string; title: string; body: RichText; tags: string[]; ... }
```

If you change a field, your editor knows immediately. No file watcher. No restart.

### 3. The escape hatch is always one prop away

For every "magical" behavior there is an explicit prop that overrides it.

| Magic                                       | Override                                |
| ------------------------------------------- | --------------------------------------- |
| Slug derived from title                     | `fields.slug({ from: 'title' })`        |
| Column auto-rendered as link                | `list.columns: [{ field, cell: Fn }]`   |
| Field rendered with default widget          | `field.widget: MyWidget`                |
| Auto-generated REST route                   | `collection.api: { handlers: {...} }`   |
| Auto-generated detail page                  | `collection.admin: { detail: MyPage }`  |

Defaults take you 90% of the way. The remaining 10% never requires forking.

### 4. The CLI is opinionated and short

```
voila init                 scaffold into an existing TanStack Start app
voila migrate              generate & apply DB migrations
voila migrate --dry        show SQL without applying
voila seed                 run seed scripts in /seed
voila studio               open the local admin without your app
voila gen client           emit a portable client (for non-TS consumers only)
voila mcp                  run the MCP server as a standalone process
voila task <id>            run a registered task once
voila cron list            show registered cron jobs
voila i18n pull            download Messages from DB → ./messages/*.json
voila i18n push            upload ./messages/*.json → DB
voila i18n status          show diff (ahead / behind / conflicts)
voila i18n watch           auto-pull on admin change (SSE)
voila i18n check           CI gate: missing translations, drift, conflicts
voila doctor               sanity-check config, env, bindings
```

That's it. No `voila plugin add`, no `voila page new`. Extensions are TypeScript imports.

### 5. Errors are actionable

Every error from `@voila/content` follows the format:

```
[voila:content] Field "posts.body" requires a richText editor adapter.

Cause:
  You set richText({ plugins: ['mention'] }) but no mention adapter is
  registered.

Fix:
  Add @voila/rich-text/mention to your config:

      import { mention } from '@voila/rich-text/mention'
      richText({ plugins: [mention({ source: 'users' })] })

Docs: https://content.voila.dev/docs/fields/rich-text#mentions
```

No silent failures, no stack traces dumped on the user, no "something went wrong".

## The 20-minute test

The DX is graded against six scenarios. Each must be doable in the time stated, by someone who knows TanStack but has never seen `@voila/content`.

| # | Scenario                                                                       | Budget   |
| - | ------------------------------------------------------------------------------ | -------- |
| 1 | Add a new collection with 6 fields and run it locally                          | 3 min    |
| 2 | Add a new field type (`color`) to a collection and see it in the admin         | 1 min    |
| 3 | Customize the admin sidebar logo & nav                                         | 2 min    |
| 4 | Add a custom dashboard widget showing today's signups                          | 5 min    |
| 5 | Add a cron job that publishes scheduled posts                                  | 5 min    |
| 6 | Deploy to Cloudflare with R2 media and D1                                      | 10 min   |

If any of these slips past budget on a clean install, it's a release blocker.

## What "extensible" actually means

Extensible is overloaded. We use it in five specific senses:

### a) Custom field widgets

```ts
fields.string({ widget: ColorPickerWidget })
```

A `Widget` is just a React component receiving `{ value, onChange, error, field }`. Use TanStack Form internals if you want.

### b) Custom admin pages

```ts
defineContent({
  pages: [
    { path: '/seo', label: 'SEO', icon: 'Globe', component: SeoPage }
  ]
})
```

Mounted under `/admin/seo`. Has access to the same context (auth, database client) as built-in pages.

### c) Dashboard widgets

```ts
defineContent({
  widgets: [
    { id: 'recent-orders', size: 'lg', component: RecentOrdersWidget }
  ]
})
```

Drag-to-reorder, persistence per user.

### d) Row & bulk actions ("buttons")

```ts
defineCollection({
  slug: 'posts',
  actions: [
    {
      id: 'translate',
      label: 'Translate',
      icon: 'Translate',
      scope: 'row',          // 'row' | 'bulk' | 'detail'
      run: async ({ doc, ctx }) => {
        await ctx.tasks.run('translate-post', { id: doc.id })
      }
    }
  ]
})
```

### e) Background work: tasks, cron, queues

```ts
defineContent({
  tasks: [
    {
      id: 'translate-post',
      input: z.object({ id: z.string() }),
      run: async ({ input, ctx }) => { /* … */ }
    }
  ],
  cron: [
    { schedule: '*/5 * * * *', task: 'publish-scheduled' }
  ]
})
```

Tasks run on Cloudflare Queues by default. Same code runs locally on a worker thread when developing.

## Local dev loop

- `bun dev` runs the playground app **and** the admin together. HMR for both.
- Schema changes are picked up by Vite without restart (HMR on the config module).
- Drizzle migrations are generated on demand by `voila migrate`; never run automatically (we don't surprise you).
- A `/__voila/debug` page in dev shows the resolved config, the live DB schema, the registered tasks/cron, and the MCP tool surface.

## Editor support

Out of the box:

- **TypeScript** strict mode, with `as const`-driven inference everywhere.
- **Biome** config shipped at the root.
- **VS Code** workspace recommendations in `.vscode/extensions.json`.
- **TanStack Devtools** for Router, Query, Form, DB all available in the admin.

## What we don't do

- We don't ship a separate "Studio" GUI for editing the schema. The schema is code. Edit it in your editor.
- We don't auto-publish breaking schema migrations. You always run `voila migrate` consciously.
- We don't hide a Node dependency. The package runs unchanged on Workers, Bun, and Node.
- We don't add a global. No `globalThis.content`. Importing is explicit.
