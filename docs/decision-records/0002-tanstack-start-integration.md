# ADR 0002 — How the admin mounts in a TanStack Start app

**Accepted 2026-05-22 · Superseded 2026-05-27 (Effect pivot → registry vending)**

## Original decision (no longer in effect)

The admin would mount via a **Vite plugin** (`@voila/content/vite`) that
auto-discovered `content.config.ts` and injected **virtual routes** + server
handlers. The plugin was to be the only integration surface; `content.handle()`
was removed.

## Why it was reversed

The Effect-based engine was removed (see [tech decisions](../tech-decisions.md)),
and with it the case for build-time magic. Virtual routes are a black box —
hard to grep, debug, or diff — which contradicts the "you own your code"
principle. A Vite plugin also locks the admin to one bundler's resolver.

## Current decision

**The admin is vended as real source files (shadcn-style registry), not
generated by a plugin.**

- `voila add <item>` copies real route files, blocks, and widgets into the app's
  `app/` tree. They are yours — greppable, editable, PR-diffable. `voila diff`
  shows drift from upstream.
- The engine mounts through a plain handler: `createRestHandler({ config,
  database }, { basePath })` wired in a normal server route file (`app/lib/
  server.ts`). No virtual routes, no plugin.
- The config entry is `defineConfig(...)` in `content.config.ts`, imported
  directly by the app, the CLI, and tests — no `virtual:` module.

**Trade-off:** a little more visible boilerplate (the vended files) in exchange
for zero build-time magic and full ownership. TanStack Start remains the
supported head; non-TanStack consumers are out of scope.
